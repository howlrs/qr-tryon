## 2. 技術選定理由

### Next.js 16 (App Router)

- **選定理由:** サーバーサイドレンダリング（SSR）とクライアントサイドレンダリング（CSR）を柔軟に使い分けられる。Route Handler でモックAPIを同一プロジェクト内に実装でき、開発初期の検証が容易。
- **App Router採用:** Route Groups によるレイアウト分離、Server Components によるパフォーマンス最適化、`[locale]` セグメントによる自然なi18n対応が可能。

### TypeScript

- **選定理由:** 型安全性によりバグを事前に防止。IDEの補完・リファクタリング支援。バックエンドのGoモデルとフロントエンドの型定義を対応付けやすい。

### Tailwind CSS 4

- **選定理由:** ユーティリティファーストでプロトタイピングが高速。CSS変数との統合によりテーマ管理が容易。コンポーネントに閉じたスタイリングで保守性が高い。

### next-intl

- **選定理由:** Next.js App Router にネイティブ対応。ルーティング統合（`createNavigation`）によりロケール管理が透過的。JSON形式のメッセージファイルで翻訳管理がシンプル。

### qrcode.react

- **選定理由:** React コンポーネントとして直接QRコードをSVG描画可能。エラー訂正レベルの設定、サイズ調整が容易。SVGからPNGへの変換でダウンロード機能を実現。

### Go（将来のバックエンド）

- **選定理由:** 高い実行速度と静的型付け。軽量な並行処理モデル。Bunを使ったORMでPostgreSQLとの接続が容易。

## 3. データフロー

### 3.1 商品閲覧フロー

```
来店客がQRコードを読み取る
    │
    ▼
ブラウザが /products/{id} を開く
    │
    ▼
Next.js Server Component がモックデータから商品を検索
    │
    ▼
ProductDetailClient（Client Component）に商品データを渡す
    │
    ▼
来店客が画像閲覧・バリエーション選択
```

### 3.2 試着/購入リクエストフロー

```
来店客が「試着する」or「購入する」をタップ
    │
    ▼
deviceId を localStorage から取得（なければ生成・保存）
    │
    ▼
POST /api/orders にリクエスト送信
    ├── deviceId, productId, variantId, type を含む
    │
    ▼
Route Handler がレート制限を確認
    ├── 1分以内の重複 → 429 を返す
    └── OK → インメモリ配列に追加、200 を返す
    │
    ▼
管理画面の AdminSidebar が 2秒ごとに GET /api/orders をポーリング
    │
    ▼
未読件数がバッジに表示される
```

### 3.3 QRコード生成フロー

```
管理者が商品一覧でQRアイコンをクリック
    │
    ▼
QRCodeModal が開く
    │
    ▼
qrcode.react が商品URLをQRコード（SVG）として描画
    ├── URL: {origin}/products/{id}
    ├── エラー訂正レベル: H（高）
    │
    ▼
「Download QR Code」クリック
    │
    ▼
SVG → Canvas → PNG に変換し、ダウンロードリンクを生成
```

## 5. レンダリング戦略

| 画面 | レンダリング | 理由 |
|------|-------------|------|
| トップページ | Server Component | 静的コンテンツ。SEO有利 |
| 商品一覧 | Server Component | モックデータの読み込みのみ |
| 商品詳細 | Server + Client | Server: データ取得、Client: インタラクション（バリエーション選択、API呼び出し） |
| 管理画面全般 | Client Component | 状態管理、イベントハンドラ、ポーリングが必要 |

## 6. 状態管理

本システムでは外部の状態管理ライブラリ（Redux等）を使用せず、React の `useState` / `useEffect` で完結している。

| 状態 | 管理場所 | 説明 |
|------|----------|------|
| 商品一覧 | `AdminProductsPage` の `useState` | モックデータで初期化、削除時にローカル更新 |
| QRモーダルの表示対象 | `AdminProductsPage` の `useState` | 選択中の商品を保持 |
| 注文一覧 | `OrdersPage` の `useState` | API からフェッチ |
| 未読件数 | `AdminSidebar` の `useState` | 2秒ポーリングで更新 |
| 選択中バリエーション | `ProductDetailClient` の `useState` | 来店客が選択したバリエーション |
| 選択中画像 | `ProductDetailClient` の `useState` | 来店客が選択した画像 |
| deviceId | `localStorage` | 端末固有のUUID。レート制限に使用 |
| フォーム入力 | `ProductForm` の複数 `useState` | 名前、説明、バリエーション配列、画像配列 |

## 7. セキュリティ考慮事項

### 現在の状態（開発フェーズ）

- **認証なし:** 管理画面に認証機構はない。URLを知っていれば誰でもアクセス可能
- **CSRF保護:** Next.js のデフォルト（SameSite Cookie）に依存
- **入力検証:** フォームの `required` 属性によるクライアントサイドのみ。サーバーサイドのバリデーションは未実装
- **レート制限:** deviceId ベースの簡易的な制限のみ（回避可能）

### 本番環境で必要な対策

| 対策 | 説明 |
|------|------|
| 管理者認証 | `manager` テーブルを使ったログイン機能の実装 |
| APIバリデーション | リクエストボディの型・値の検証（サーバーサイド） |
| HTTPS強制 | 全通信のTLS暗号化 |
| CORS設定 | APIのオリジン制限 |
| レート制限強化 | IPベース + deviceId での二重制限 |
| SQLインジェクション対策 | ORM / パラメタライズドクエリの使用（Go移行時） |
| 画像URL検証 | 許可されたドメインのみ受け付ける |

## 8. 将来の拡張計画

### フェーズ2: Go バックエンド構築

現在 Next.js Route Handler で実装しているAPIを Go に移行する。

```
frontend (Next.js)
    │
    │  HTTP REST
    ▼
backend (Go)
    ├── Handlers   ← HTTPリクエスト処理
    ├── Services   ← ビジネスロジック
    ├── Repositories ← データアクセス
    └── Models     ← データ構造定義
```

**移行手順:**
1. Go プロジェクトを作成し、現在の Route Handler の仕様をそのまま実装
2. Next.js 側の API 呼び出し先を Go サーバーに変更
3. Route Handler を削除

### フェーズ3: データベース接続

- PostgreSQL をデータストアとして導入
- Bun (Go ORM) でモデル定義・マイグレーション管理
- インメモリストアを永続化ストレージに置き換え

### フェーズ4: 機能拡充（検討中）

| 機能 | 説明 |
|------|------|
| 管理者認証 | ログイン/ログアウト、セッション管理 |
| リアルタイム通知 | ポーリングからWebSocketへの移行 |
| 画像アップロード | 外部URL入力からファイルアップロードへ |
| 在庫自動更新 | 購入リクエスト承認時に在庫を自動減算 |
| 売上レポート | 期間別の売上・試着リクエスト統計 |
| 多店舗対応 | 店舗ごとのデータ分離・管理者権限 |

## 9. パフォーマンス考慮事項

### 現在の最適化

- **Server Components:** 公開画面のデータ取得をサーバーサイドで実行し、クライアントへの JavaScript 配信を最小化
- **画像最適化:** `next/image` による自動最適化（フォーマット変換、遅延読み込み、サイズ調整）
- **フォント最適化:** `next/font` による Google Fonts の自動最適化（セルフホスティング）

### 将来の改善点

| 項目 | 対策 |
|------|------|
| ポーリング負荷 | WebSocket または Server-Sent Events に移行 |
| モックデータ | データベースからの効率的なクエリに置き換え |
| バンドルサイズ | 動的インポート（`dynamic`）の活用 |
| キャッシュ | API レスポンスのキャッシュ戦略（ISR / SWR） |
